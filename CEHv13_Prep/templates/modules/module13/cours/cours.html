1. Concepts des serveurs web

Architecture web fondamentale

  1. Client: Navigateur web (Chrome, Firefox)
  2. Serveur Web: Apache, Nginx, IIS
  3. Serveur d'Application: Node.js, Tomcat, Django
  4. Base de Données: MySQL, PostgreSQL, MongoDB

Serveurs web populaires

Serveur Part de marché Plateforme
Apache 31.4% Multiplateforme
Nginx 33.6% Multiplateforme
IIS 19.7% Windows
LiteSpeed 8.2% Multiplateforme

Composants vulnérables

  • Modules/extensions
  • Services (FTP, SSH)
  • Fichiers de configuration
  • Journaux d'accès
  • Scripts CGI

Statistiques de sécurité

  • 70% des serveurs web ont au moins une vulnérabilité
  • Apache représente 48% des serveurs compromis
  • Les attaques web ont augmenté de 56% en 2023
  • Coût moyen d'une violation: $4.35 millions

2. Attaques de serveurs web

Attaques courantes

Attaque Description Impact
Injection SQL Exécution de commandes SQL non autorisées Élevé
Cross-Site Scripting (XSS) Exécution de scripts malveillants côté client Moyen
Directory Traversal Accès à des fichiers en dehors du répertoire web Élevé
Server-Side Request Forgery (SSRF) Forcer le serveur à effectuer des requêtes internes Critique
XML External Entity (XXE) Exploitation des analyseurs XML Élevé
Exemple: Directory Traversal
http://example.com/../../../../etc/passwd

Permet d'accéder au fichier système /etc/passwd

Attaques spécifiques aux serveurs

  • Apache: Mod_status exposé, .htaccess vulnérable
  • Nginx: Mauvaise configuration alias, traversée de chemin
  • IIS: RCE via WebDAV, vulnérabilités ISAPI
  • Tomcat: Manager App non sécurisé, déploiement WAR malveillant

Attaques de déni de service

  • Slowloris: Maintient les connexions ouvertes
  • HTTP Flood: Inonde le serveur de requêtes
  • RUDY: Envoi lent de corps POST volumineux
  • Apache Killer: Exploite la compression

Démonstration: Injection SQL

  1. Repérer un paramètre vulnérable: product.php?id=1
  2. Tester: id=1' (erreur SQL?)
  3. Exploiter: id=1' UNION SELECT username, password FROM users--
  4. Extraire les données sensibles

3. Méthodologie d'attaque

Étapes du hacking de serveurs web

  1. Reconnaissance: Collecte d'informations
  2. Scanning: Identification des services et ports
  3. Énumération: Découverte des ressources
  4. Exploitation: Abus des vulnérabilités
  5. Escalade de privilèges: Obtention d'accès élevés
  6. Maintenance de l'accès: Installation de backdoors
  7. Effacement des traces: Suppression des logs

Techniques de reconnaissance

  • WHOIS: Information sur le domaine
  • DNS Enumeration: Découverte des sous-domaines
  • Search Engine Hacking: Dorking (site:example.com filetype:pdf)
  • Wayback Machine: Archives historiques
  • Headers HTTP: Identification des technologies

Scanning avec Nmap

nmap -sV -sC -p 80,443,8080 example.com
nmap --script http-enum,http-vuln* example.com
nmap -sV --script banner example.com

Options importantes:

  • -sV: Détection des versions
  • -sC: Exécution des scripts par défaut
  • -p: Ports spécifiques

Énumération des serveurs web

  • Découverte des répertoires: gobuster dir -u https://example.com -w wordlist.txt
  • Recherche de sous-domaines: sublist3r -d example.com
  • Analyse des robots.txt et sitemap.xml
  • Inspection des certificats SSL
  • Recherche de fichiers exposés: backup.zip, .git, etc.
Exploitation réussie

Après exploitation, les attaquants installent généralement:

  • Web shells (PHP, ASP, JSP)
  • Backdoors persistantes
  • Rootkits niveau noyau
  • Scripts de minage de cryptomonnaie

Outils d'énumération

4. Contre-mesures

Durcissement des serveurs

  • Configuration sécurisée: Suppression des en-têtes serveur
  • Principle of Least Privilege: Limitation des permissions
  • Segmentation réseau: Isolation des composants
  • Chiffrement: TLS 1.3 avec chiffrements forts
  • Désactivation des services inutiles: Réduction de la surface d'attaque

Protections logicielles

  • Web Application Firewall (WAF)
  • Systèmes de détection/prévention d'intrusion (IDS/IPS)
  • Antivirus serveur
  • Contrôle d'intégrité des fichiers
  • Journalisation centralisée

Meilleures pratiques de sécurité

  • Validation des entrées utilisateur
  • Échappement des sorties
  • Requêtes préparées pour les bases de données
  • Mise en œuvre de Content Security Policy (CSP)
  • Authentification multi-facteurs (MFA)
  • Rotation régulière des clés et certificats

Configuration sécurisée Apache

ServerTokens Prod
ServerSignature Off
TraceEnable Off
Header unset X-Powered-By
Options -Indexes -Includes -ExecCGI
Configuration sécurisée Nginx
server_tokens off;
add_header X-Content-Type-Options nosniff;
add_header X-Frame-Options SAMEORIGIN;
add_header X-XSS-Protection "1; mode=block";
add_header Content-Security-Policy "default-src 'self'";

5. Gestion des correctifs

Cycle de gestion des correctifs

  1. Identification: Détection des vulnérabilités
  2. Évaluation: Analyse des risques
  3. Acquisition: Obtention des correctifs
  4. Test: Validation en environnement de test
  5. Déploiement: Mise en production
  6. Vérification: Confirmation du succès
  7. Documentation: Enregistrement des changements

Types de correctifs

Type Fréquence Criticité
Sécurité critique Immédiate (24-48h) Élevée
Sécurité importante 1-2 semaines Moyenne
Mises à jour fonctionnelles 1-3 mois Faible

Conséquences du non-patching

  • 70% des violations exploitent des vulnérabilités connues
  • Temps moyen de correction: 102 jours
  • 60% des violations liées à des correctifs non appliqués
  • Coût moyen d'une violation évitable: $1.2 million

Outils de gestion des correctifs

  • Windows: WSUS, SCCM
  • Linux: Spacewalk, Foreman, Landscape
  • Multiplateforme: Ansible, Puppet, Chef
  • Cloud: AWS Systems Manager, Azure Update Management

Meilleures pratiques

  • Inventaire complet des actifs
  • Analyse continue des vulnérabilités
  • Classification par criticité
  • Environnements de test dédiés
  • Automatisation du déploiement
  • Surveillance post-déploiement

Processus automatisé avec Ansible

- name: Apply security patches
  hosts: webservers
  become: yes
  tasks:
    - name: Update apt cache (Debian)
      apt:
        update_cache: yes
      when: ansible_os_family == 'Debian'

    - name: Apply security updates (Debian)
      apt:
        upgrade: dist
        update_cache: yes
      when: ansible_os_family == 'Debian'

    - name: Apply security updates (RHEL)
      yum:
        name: '*'
        security: yes
        state: latest
      when: ansible_os_family == 'RedHat'

    - name: Reboot server if required
      reboot:
        msg: "Reboot initiated by Ansible for security updates"
        connect_timeout: 5
        reboot_timeout: 600
        pre_reboot_delay: 0
        post_reboot_delay: 30
        test_command: uptime
