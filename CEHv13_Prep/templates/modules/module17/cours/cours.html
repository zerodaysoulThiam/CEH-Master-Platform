1. Hacking des plateformes mobiles

Vecteurs d'attaque sur les plateformes mobiles

L'utilisation croissante des appareils mobiles, notamment dans le cadre des politiques BYOD (Bring Your Own Device), a élargi la surface d'attaque. Les menaces peuvent venir de plusieurs couches :

  • Appareil : attaques de type phishing, malwares
  • Réseau : écoute sur les connexions Wi-Fi publiques, interception de données
  • Infrastructure distante : failles sur les services utilisés (API, services cloud, etc.)

Principales menaces selon l’OWASP Top 10 Mobile

  1. Mauvaise utilisation de la plateforme : non-respect des bonnes pratiques de sécurité (permissions, API, empreintes digitales mal configurées).
  2. Stockage de données non sécurisé : données sensibles non chiffrées ou accessibles via l’exploration du système de fichiers.
  3. Communication non sécurisée : échanges en clair (absence de SSL/TLS) ou mauvaise gestion des certificats.
  4. Authentification insuffisante : mots de passe faibles, absence de gestion de session sécurisée, TouchID mal configuré.
  5. Cryptographie faible : utilisation d’algorithmes obsolètes, clés mal protégées ou solutions de chiffrement maison peu fiables.
  6. Autorisation insuffisante : accès à des ressources protégées via des IDOR, faibles vérifications côté serveur.
  7. Failles de qualité du code : injection de code, débordement mémoire, exécution de code distant.
  8. Modification de code (code tampering) : altération des APK/iOS, injection de ressources malveillantes.
  9. Ingénierie inverse : décompilation des applications pour voler des secrets (API keys, logique métier).
  10. Fonctionnalités inutiles : présence de code de test ou de fonctions cachées permettant d’obtenir des privilèges élevés.

Menaces courantes

  • Malwares et rootkits
  • Perte ou vol de données
  • Altération ou exfiltration des données

Vulnérabilités fréquentes

  • Applications malveillantes ou mal codées
  • Demandes de permissions excessives
  • Chiffrement faible ou inexistant
  • Mises à jour absentes ou tardives (OS ou app)
  • Rooting/Jailbreaking non contrôlés
  • Accès physique non sécurisé

Android

API d’administration de l’appareil

Permet aux entreprises de créer des applications capables de gérer les paramètres de sécurité du téléphone, comme la politique de mot de passe, l’effacement à distance, etc.

Rooting

Le rooting permet d’obtenir un accès administrateur complet sur un appareil Android. Cela peut être utile pour les tests de sécurité, mais représente un grand risque s’il est fait à l’insu de l’utilisateur. Exemple : SuperOneClick.

iOS

Jailbreaking

Similaire au rooting Android, le jailbreaking permet de supprimer les restrictions imposées par Apple sur les appareils iOS.

Types d’exploits

  • Userland : accès utilisateur seulement
  • iBoot : accès système et utilisateur
  • BootROM : accès complet, non corrigeable sur certains appareils

Techniques

  • Untethered : fonctionne sans PC au redémarrage
  • Tethered : nécessite un PC pour redémarrer
  • Semi-untethered : redémarrage possible, mais fonctionnalités limitées
  • Semi-tethered : comme le précédent, mais possibilité de relancer le jailbreak depuis une app

Autres plateformes

Windows Phone

Utilise le noyau Windows NT. Les versions modernes incluent un sandboxing d'applications, une gestion distante des appareils, et un support du code natif (C++).

BlackBerry OS

Supporte Java ME (MIDP 1.0/2.0). Mises à jour via le système OTASL (Over The Air Software Loading).

Gestion des appareils mobiles (MDM)

MDM est essentiel pour sécuriser, surveiller et gérer les appareils mobiles dans un contexte professionnel.

Fonctions principales

  • Verrouillage automatique après échec d’authentification
  • Application de politiques de mot de passe fort
  • Détection des tentatives de compromission
  • Chiffrement des données
  • Mise en œuvre de politiques de prévention contre la perte de données (DLP)

Méthodes de déploiement

  • Sur site : contrôle total mais nécessite de l'expertise interne
  • Dans le cloud : géré par un prestataire, plus simple à maintenir

Politique BYOD (Bring Your Own Device)

Autoriser les employés à utiliser leurs propres appareils présente des avantages, mais aussi des risques.

Bonnes pratiques BYOD

  • Limiter les types d’appareils autorisés
  • Exiger des mots de passe forts
  • Contrôler les données accessibles
  • Limiter les applications autorisées
  • Imposer un chiffrement du stockage

Bonnes pratiques de sécurité mobile

  • Éviter l’upload automatique des fichiers
  • Effectuer des audits de sécurité réguliers des applications
  • Désactiver le Bluetooth quand inutile
  • Limiter les apps utilisant le GPS
  • Éviter les réseaux Wi-Fi ouverts
  • Installer les apps uniquement depuis les sources officielles
  • Utiliser un mot de passe complexe
  • Utiliser une solution MDM
  • Mettre régulièrement à jour le système
  • Interdire le rooting/jailbreaking
  • Chiffrer le stockage interne
  • Faire des sauvegardes régulières
  • Appliquer des politiques de sécurité cohérentes

2. Hacking du Système d’Exploitation Android

Android, en tant que système d’exploitation open-source dominant le marché mobile mondial, constitue une cible privilégiée pour les cyberattaques. Cette section explore les différentes méthodes d'exploitation des failles Android, les techniques de rooting, les outils d'analyse, ainsi que les contre-mesures adaptées, conformément au programme CEH v13.

Principaux vecteurs d'attaque sur Android

  • Applications malveillantes : Logiciels déguisés en applications légitimes pour exécuter des actions non autorisées.
  • Permissions abusives : Demandes excessives d'autorisations sensibles (SMS, micro, stockage, etc.).
  • Wi-Fi non sécurisé : Interception de données via des réseaux publics ou falsifiés (Evil Twin).
  • Phishing par SMS (SMiShing) : Liens piégés envoyés par SMS pour inciter à télécharger un malware ou voler des identifiants.
  • Exploitation de vulnérabilités système : Utilisation de failles connues pour exécuter du code arbitraire ou obtenir un accès root.

Techniques de rooting

Le rooting permet à un attaquant d’obtenir un accès administrateur complet sur un appareil Android. Cela permet l’installation de malwares profonds ou la désactivation des mécanismes de sécurité.

  • Rooting basé sur des exploits : Exploitation de vulnérabilités dans le noyau ou les services système (ex. Dirty Cow).
  • Outils populaires : Magisk, KingoRoot, SuperSU.
  • Risques : Suppression des sécurités natives, incompatibilité logicielle, ouverture à d’autres attaques (keyloggers, spyware).

Outils d’exploitation et d’analyse Android

  • Metasploit : Génération de payloads Android (`apk`) pour l’ingénierie sociale ou l’exploitation réseau.
  • APKTool : Décompilation et reconstitution d’applications Android pour audit et modification de code.
  • Drozer : Cadre d'évaluation de la sécurité des applications Android (tests d’intent, activités exportées, etc.).
  • ADB (Android Debug Bridge) : Interface en ligne de commande pour contrôler un appareil Android connecté (push, shell, install...).
  • Frida : Instrumentation dynamique pour manipuler les fonctions internes d'une application à l'exécution.
  • MobSF (Mobile Security Framework) : Outil complet d’analyse statique et dynamique de fichiers APK.

Vulnérabilités courantes (OWASP Mobile Top 10)

Rang Catégorie OWASP Description détaillée Exemple concret Impact potentiel Mesures de prévention
M1 Improper Platform Usage Mauvaise utilisation des fonctionnalités natives du système (WebView, intents, permissions...) Une app utilise WebView sans filtrer les URL : redirection vers un site malveillant Prise de contrôle à distance, exécution de code Utiliser correctement les API ; filtrer les URL et intents
M2 Insecure Data Storage Données sensibles mal stockées (non chiffrées) dans fichiers, préférences, base SQLite... Mots de passe en clair dans SharedPreferences ou SQLite Vol d’identifiants ou de données bancaires Utiliser le Keystore / Keychain ; chiffrer localement les données
M3 Insecure Communication Données échangées sans chiffrement adéquat (ex. : HTTP au lieu de HTTPS) Transmission d’infos via HTTP interceptable par un proxy Interception de données (attaque Man-in-the-Middle) Utiliser TLS 1.2+ avec certificats valides
M4 Insecure Authentication Authentification contournable, sessions non sécurisées ou non expirables Jeton d’authentification réutilisable sans expiration Usurpation d’identité, contournement de login Implémenter des sessions expirables et vérifications côté serveur
M5 Insufficient Cryptography Mauvais algorithmes de chiffrement (ex. : MD5), clés en dur, pas de salage Utilisation de MD5 ou AES avec une clé codée en dur Décryptage facile, fuite de données sensibles Utiliser AES-256 avec salage ; stocker les clés dans un Keystore

M7 Client Code Quality Bugs, messages d’erreurs révélant des infos sensibles, gestion d’erreurs faible Chemins internes ou logs affichés à l’écran en cas d’erreur Exposition d’infos utiles à un attaquant Filtrer les erreurs, nettoyer les logs, tester le code client
M8 Code Tampering Modification du code de l’application après décompilation (injection malicieuse) Injection d’un keylogger dans un APK modifié Vol de données utilisateurs, comportement modifié Signer l’APK, vérifier l’intégrité à l’exécution
M9 Reverse Engineering Décompilation et analyse du code (récupération de secrets, algos, API...) Analyse d’un APK avec JADX pour extraire des clés codées en dur Vol de secrets, reproduction d’algorithmes propriétaires Obfuscation du code, éviter les secrets codés en dur
M10 Extraneous Functionality Fonctionnalités de debug ou test laissées accessibles en production Console ou page admin accessible via une URL cachée Exposition de fonctions internes critiques Nettoyer le code avant déploiement ; désactiver le debug

Contre-mesures et bonnes pratiques

  • Chiffrer toutes les données sensibles stockées ou transmises.
  • Utiliser des certificats SSL/TLS valides et correctement implémentés.
  • Éviter le stockage de secrets (API keys, tokens) en dur dans le code source.
  • Activer la détection de rooting/jailbreak pour bloquer les appareils compromis.
  • Implémenter la validation côté serveur pour toute authentification/autorisation.
  • Utiliser ProGuard ou R8 pour l’obfuscation du code et la réduction de surface d’attaque.
  • Effectuer des audits réguliers (static/dynamic) de l'application via des outils comme MobSF.
  • Réduire les permissions requises au strict minimum.
  • Signer numériquement les APKs et vérifier l’intégrité avant exécution.

Conclusion

Maîtriser les techniques de hacking Android est essentiel pour anticiper les menaces qui ciblent les utilisateurs mobiles. Le CEH v13 forme les professionnels à identifier les faiblesses d’un environnement Android et à mettre en œuvre les bonnes pratiques de défense. Comprendre ces vecteurs d’attaque permet non seulement d’évaluer les risques, mais aussi de renforcer la sécurité mobile des infrastructures professionnelles.

4. Hacking iOS

Le système d'exploitation iOS d'Apple est réputé pour sa sécurité intégrée, mais il n’est pas invulnérable. Un testeur d'intrusion certifié CEH doit comprendre les vulnérabilités spécifiques de cette plateforme mobile fermée. Cette section couvre les techniques fondamentales et avancées utilisées dans le cadre de l’éthique du hacking mobile iOS.

🔒 Architecture de sécurité iOS

  • Sandboxing : Chaque application est isolée des autres pour limiter les dommages potentiels.
  • Code Signing : Toute app doit être signée numériquement pour être exécutée.
  • ASLR : Address Space Layout Randomization rend l'exploitation mémoire plus difficile.
  • Touch/Face ID & Secure Enclave : Fournissent un niveau élevé de sécurité biométrique.

🔍 Techniques de reconnaissance sur iOS

  • Analyse du binaire IPA : Extraction avec unzip puis analyse avec class-dump ou Hopper.
  • Inspection des chaînes : Utilisation de strings pour rechercher des clés API ou mots de passe codés en dur.
  • Reverse Engineering : Désassemblage avec Hopper ou Ghidra.

🧪 Techniques d’attaque spécifiques à iOS

  • Jailbreak-based Attacks : Gain de privilèges root pour injecter ou intercepter du code.
  • Interception SSL : Si SSL pinning est mal implémenté, utilisation d’outils comme mitmproxy ou Burp Suite.
  • Bypass Authentification : Modification de la logique d’authentification (patch d’IPA, injection via Frida).
  • Injection dynamique : Manipulation du comportement runtime avec Frida ou Cycript.
  • Analyse de Keychain : Extraction de données sensibles stockées dans le porte-clés iOS.

🔧 Outils utilisés pour le hacking iOS

  • Frida : Hooking dynamique d’applications sans recompilation
  • Objection : Framework post-exploitation pour apps iOS
  • Cycript : Manipulation d’objets en live dans une app Objective-C
  • Burp Suite + iOS proxy : Interception de trafic (HTTP/HTTPS)
  • MobSF : Analyse statique et dynamique automatisée des APK/IPA
  • Hopper Disassembler / Ghidra : Reverse engineering des binaires iOS

⚠️ Vulnérabilités fréquentes (OWASP Mobile Top 10)

Voir ci-dessous un tableau OWASP Mobile Top 10 détaillé appliqué à iOS :

🧰 Jailbreaking et pentest

Le jailbreak est souvent nécessaire pour obtenir un shell root, installer des outils de pentest, et interagir avec les apps. Les outils comme Checkra1n, Taurine, ou Palera1n sont utilisés selon la version iOS.

Risques liés au jailbreak :

  • Désactivation des protections natives d’iOS
  • Ouverture des ports SSH par défaut
  • Risque d’installation d’apps malveillantes non signées

🛡️ Contre-mesures & Bonnes pratiques pour développeurs iOS

  • Activer App Transport Security (ATS) pour forcer le HTTPS
  • Implémenter le SSL Pinning (et empêcher sa désactivation)
  • Ne pas stocker de données sensibles dans UserDefaults ou plist
  • Utiliser Keychain avec les bons attributs (ex : kSecAttrAccessibleWhenUnlocked)
  • Obfuscation du code natif (ex : Swift Shield)
  • Désactiver les logs en production

4. Mobile Device Management (MDM)

La gestion des appareils mobiles (MDM) est une composante essentielle de la cybersécurité mobile en entreprise. Elle permet aux administrateurs de contrôler, sécuriser, et appliquer des politiques sur les smartphones et tablettes utilisés dans un environnement professionnel. Cette section est cruciale pour comprendre comment protéger et évaluer la sécurité des environnements mobiles.

🔧 Qu’est-ce qu’un MDM ?

Le MDM est une solution logicielle qui permet :

  • Le déploiement et la configuration d’applications mobiles
  • La gestion à distance des paramètres des appareils
  • Le chiffrement et la protection des données
  • Le suivi et l’inventaire des appareils
  • Le verrouillage ou l’effacement d’un appareil à distance

🏢 Cas d’usage en entreprise

  • Appliquer les politiques de sécurité sur les appareils BYOD (Bring Your Own Device)
  • Empêcher l’installation d’applications non autorisées
  • Forcer le chiffrement de stockage
  • Assurer la conformité avec des normes comme ISO 27001 ou HIPAA

🛡️ Composants de sécurité d’un MDM

  • Chiffrement des données : stockage et transmission des données de manière sécurisée
  • Gestion des certificats : pour les connexions VPN ou Wi-Fi
  • Contrôle des applications : listes blanches/noires d’apps
  • Authentification renforcée : par biométrie, OTP ou SSO
  • Conteneurisation : séparation des données personnelles et professionnelles

🧰 Outils et solutions MDM populaires

  • Microsoft Intune – Intégré à Microsoft 365, gestion multi-plateforme
  • VMware Workspace ONE – Support avancé de BYOD et sécurité Zero Trust
  • Jamf Pro – Spécialisé dans les appareils Apple
  • IBM MaaS360 – Solution cloud complète pour iOS, Android et Windows
  • MobileIron (Ivanti) – Très utilisé dans les environnements sensibles

🎯 Attaques potentielles contre les MDM

Un pentesteur doit connaître les vecteurs d’attaque suivants :

  • Credential Harvesting : Vol des identifiants d’accès au panneau d’administration MDM
  • Man-in-the-Middle : Si le MDM communique sans TLS sécurisé
  • Reverse Engineering : Des apps MDM mal protégées pour trouver des clés d’API
  • Faux appareils : Inscription d’un appareil pirate pour exfiltrer des apps ou configurations
  • Escalade de privilèges : Via une faille dans l’agent MDM (ex : accès root via vulnérabilité)

📉 Vulnérabilités courantes dans les MDM

  • Accès aux API sans authentification forte
  • Stockage des jetons et credentials en clair dans l’application
  • Utilisation de certificats non valides ou expirés
  • Mise à jour logicielle non sécurisée
  • Mauvaise configuration des droits d’accès (RBAC)

✅ Bonnes pratiques de sécurité MDM

  • Activer le chiffrement natif sur tous les appareils
  • Imposer des mots de passe forts et l’authentification multi-facteurs
  • Utiliser des profils de configuration sécurisés
  • Limiter les droits d’administrateur dans la console MDM
  • Surveiller les logs et alertes pour repérer les comportements anormaux
  • Appliquer régulièrement les mises à jour de sécurité du MDM et des OS mobiles

5. Mobile Security Guidelines & Tools

Cette section présente les lignes directrices fondamentales pour la sécurisation des dispositifs mobiles, ainsi que les outils standards utilisés pour l'analyse de la sécurité mobile. Elle se concentre exclusivement sur les aspects techniques exigés dans le cadre de la certification CEH v13.

📖 Mobile Security Guidelines (Directives de sécurité mobile)

  • Application Sandboxing : Isolation stricte des processus et des données entre les applications.
  • Secure Storage : Utilisation de systèmes de stockage sécurisés (Keychain sur iOS, Keystore sur Android).
  • Secure Communication : Implémentation de TLS 1.2+ avec vérification des certificats et du hostname.
  • Minimal Permissions : Attribution du minimum de permissions nécessaires aux applications.
  • Input Validation : Vérification et filtrage des données entrantes côté client et serveur.
  • Code Obfuscation : Obfuscation du code source pour ralentir l’ingénierie inverse.
  • Root/Jailbreak Detection : Détection d’appareils compromis pour bloquer ou restreindre l’accès.
  • Session Management : Gestion sécurisée des tokens et expiration automatique des sessions.
  • Logging Restrictions : Pas de journalisation de données sensibles sur l’appareil ou dans le cloud.
  • Secure Updates : Mécanismes de mise à jour sécurisés et signés numériquement.

🧪 Outils pour l’analyse de la sécurité mobile

1. 📱 Outils d’analyse statique (Static Analysis)

  • MobSF (Mobile Security Framework) – Analyse statique, dynamique et API scanning pour Android/iOS.
  • QARK – Quick Android Review Kit : détection automatique de vulnérabilités sur APKs.
  • Androguard – Analyse de fichiers DEX/APK, reverse engineering de code Dalvik.
  • ClassyShark – Explorateur de classes Java pour applications Android.
  • apktool – Décompilation de fichiers APK en smali pour analyse manuelle.

2. ⚙️ Outils d’analyse dynamique (Dynamic Analysis)

  • Frida – Framework d’injection de code pour manipuler le comportement d’une application en temps réel.
  • Burp Suite – Proxy HTTP pour intercepter et modifier le trafic réseau mobile.
  • Objection – Framework pour l’analyse runtime sur appareils rootés ou jailbreakés.
  • Xposed Framework – Pour injecter des modules personnalisés dans les apps Android.
  • Wireshark – Analyse du trafic réseau mobile pour identifier les échanges non chiffrés.

3. 🧩 Outils pour reverse engineering

  • Ghidra – Décompilation et analyse de binaires iOS/Android.
  • Hopper Disassembler – Pour l’analyse des binaires iOS Objective-C/Swift.
  • IDA Pro – Analyse avancée du code binaire (payant).
  • dex2jar – Conversion de fichiers DEX vers JAR pour lecture dans JD-GUI.
  • JD-GUI – Lecture du code Java extrait depuis dex2jar.

4. 🔒 Outils spécialisés iOS

  • Keychain Dumper – Extraction des données stockées dans le Keychain.
  • Cycript – Injection de code Objective-C dans les apps iOS.
  • class-dump – Extraction des interfaces Objective-C depuis des binaires iOS.
  • iRET (iOS Reverse Engineering Toolkit) – Suite d’outils pour l’analyse d’applications iOS.
  • SSL Kill Switch 2 – Désactivation de la vérification SSL pour l’analyse MITM.

📌 Résumé des objectifs d’apprentissage

  • Connaître les lignes directrices fondamentales pour sécuriser les applications et appareils mobiles.
  • Savoir identifier et évaluer les vulnérabilités dans les applications mobiles.
  • Maîtriser l’usage d’outils statiques et dynamiques pour tester la sécurité mobile.
  • Apprendre les méthodes de reverse engineering d’applications mobiles Android et iOS.
