1. Cloud Computing concepts

Le cloud computing permet l'accès à la demande à des ressources informatiques via Internet, sans avoir à gérer l'infrastructure sous-jacente.

Avant le cloud computing

  • Les données et applications étaient hébergées localement (on-premises).
  • Les systèmes étaient centralisés, isolés et segmentés.

Définition selon le NIST

Le NIST définit le cloud computing comme un modèle permettant un accès pratique, à la demande, via un réseau à un ensemble partagé de ressources informatiques configurables.

Voir la définition complète

Caractéristiques essentielles

  • Self-service à la demande : les utilisateurs peuvent provisionner des ressources automatiquement sans intervention humaine.
  • Accès réseau étendu : disponible sur divers appareils via Internet.
  • Mutualisation des ressources : ressources partagées entre plusieurs clients avec isolement logique.
  • Élasticité rapide : possibilité d’augmenter ou réduire rapidement les ressources selon les besoins.
  • Service mesuré : la consommation est surveillée et facturée à l’usage.

Modèles de service

IaaS (Infrastructure as a Service)

Fournit des ressources de base comme des machines virtuelles, le stockage et les réseaux. L'utilisateur gère le système d’exploitation et les applications.

PaaS (Platform as a Service)

Fournit un environnement de développement complet géré par le fournisseur. L'utilisateur se concentre sur le développement sans gérer l'infrastructure.

SaaS (Software as a Service)

Fournit des applications hébergées et accessibles via un navigateur. L'utilisateur n’a aucun contrôle sur l’infrastructure.

Autres modèles

  • IDaaS : services d’authentification gérés comme SSO, MFA.
  • SECaaS : services de sécurité comme l’antivirus, la détection d'intrusion.
  • CaaS : services de gestion de conteneurs et de clusters.
  • FaaS : exécution de fonctions sans gestion de serveurs (ex : AWS Lambda).

Responsabilités partagées

  • On-premises : tout est géré par l’entreprise.
  • IaaS : le fournisseur gère le matériel, l’entreprise gère les logiciels.
  • PaaS : le fournisseur gère aussi le système et les plateformes.
  • SaaS : tout est géré par le fournisseur sauf la configuration de l’application.

Modèles de déploiement

  • Cloud privé : dédié à une seule organisation, sur site ou chez un prestataire.
  • Cloud public : partagé via Internet, hébergé par un fournisseur.
  • Cloud communautaire : partagé entre plusieurs organisations ayant les mêmes objectifs.
  • Cloud hybride : combinaison de plusieurs clouds (public, privé...)
  • Multi-cloud : utilisation de plusieurs fournisseurs cloud différents.

Acteurs du cloud (selon NIST)

  • Consommateur : utilisateur final des services cloud.
  • Fournisseur : entité qui fournit les services cloud.
  • Auditeur : effectue des évaluations indépendantes.
  • Broker : intermédiaire entre fournisseurs et consommateurs.
  • Transporteur : assure la connectivité réseau.

Séparation des responsabilités

Permet de mieux gérer la sécurité :

  • Propriétaire des données : décide de leur utilisation.
  • Gestionnaire : assure la protection et la maintenance.
  • Risque : si une même personne cumule les deux rôles.

Avantages du cloud

  • Économies de coûts.
  • Élasticité et flexibilité.
  • Haute disponibilité.
  • Accès rapide à l’innovation.
  • Meilleure reprise après sinistre.

Inconvénients du cloud

  • Dépendance au fournisseur.
  • Problèmes de sécurité et de conformité.
  • Moins de contrôle sur l’infrastructure.
  • Dépendance à la connexion Internet.
  • Possibilité de verrouillage contractuel (vendor lock-in).

Réglementations

  • FedRAMP : régulation américaine du cloud.
  • PCI DSS : sécurité des données bancaires, y compris dans le cloud.

Accords de niveau de service (SLA)

Les SLA définissent les engagements du fournisseur envers le client (temps de disponibilité, support, sécurité...)

2. Container Technology

La technologie des conteneurs révolutionne le déploiement des applications en permettant leur exécution dans des environnements isolés, reproductibles et portables. Contrairement à la virtualisation classique, elle repose sur un partage du noyau de l’OS et s’appuie sur des outils comme Docker, Kubernetes, Podman, etc.

Introduction

Un conteneur est une unité standard de logiciel qui emballe le code et toutes ses dépendances pour que l’application s’exécute rapidement et de manière fiable dans différents environnements. Les conteneurs sont plus légers que les machines virtuelles car ils partagent le noyau de l’OS hôte.

Container vs Virtual Machine

Aspect Conteneur Machine Virtuelle (VM)
Isolation Processus isolés, même noyau OS OS complet isolé, avec hyperviseur
Démarrage En quelques secondes En quelques minutes
Taille Quelques Mo à centaines de Mo Plusieurs Go
Performances Meilleure (moins de surcharge) Plus lente (hyperviseur)
Utilisation principale Microservices, CI/CD Exécution complète de systèmes

Architecture d’un conteneur

  • Application : Code, binaires, bibliothèques
  • Conteneur Runtime : ex. Docker, containerd
  • Hôte OS : système d’exploitation supportant le runtime
  • Noyau Linux partagé : pas de système invité (contrairement aux VM)

Avantages des conteneurs

  • Portabilité
  • Isolation
  • Rapidité
  • Efficacité
  • Scalabilité
  • Reproductibilité

Docker : Le standard de facto

Composants Docker

  • Docker Engine
  • Docker CLI
  • Dockerfile
  • Docker Image
  • Docker Container
  • Docker Hub

Commandes de base

docker build -t nom_image .
docker run -d -p 8080:80 nom_image
docker ps
docker exec -it nom_conteneur bash
docker stop nom_conteneur

Dockerfile : Structure d’une image

FROM python:3.11
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
CMD ["python", "app.py"]
  • FROM : image de base
  • WORKDIR : répertoire de travail
  • COPY : copie des fichiers
  • RUN : exécution de commandes à la création de l’image
  • CMD : commande exécutée au lancement du conteneur

Orchestration de conteneurs

Kubernetes (K8s)

  • Pod : plus petite unité déployable
  • Service : abstraction réseau
  • Deployment : gère la réplication et la mise à jour
  • Ingress : contrôle l'accès externe

Alternatives

  • Docker Swarm
  • Nomad (HashiCorp)
  • OpenShift

Container Registry

  • Public : Docker Hub, GitHub Container Registry, Quay.io
  • Privé : AWS ECR, GitLab, Harbor

Sécurité des conteneurs

Risques

  • Images malveillantes ou vulnérables
  • Conteneurs mal configurés
  • Fuite de données

Contremesures

  • Scanner les images : Trivy, Clair
  • Signer les images : Notary, Cosign
  • Utiliser des images minimales : Alpine, Distroless
  • Limiter les permissions : rootless containers
  • Isoler avec namespaces et cgroups

Container as a Service (CaaS)

  • AWS ECS / EKS
  • Azure AKS
  • Google GKE

Serverless vs Containers

Aspect Conteneur Serverless (FaaS)
Démarrage Rapide, mais pas instantané Instantané (cold start possible)
Contrôle Complet sur l'environnement Moins de contrôle
Durée d’exécution Persistante Éphémère
Coût Par ressource utilisée Par appel ou consommation exacte

Cas d’usage des conteneurs

  • Microservices
  • Développement multi-environnements
  • CI/CD
  • Tests automatisés
  • Migration vers le cloud
  • Éducation et formation

Meilleures pratiques

  • Construire des images légères
  • Éviter d'exécuter en tant que root
  • Utiliser .dockerignore
  • Réduire le nombre de couches
  • Versionner les images
  • Automatiser les tests

Outils complémentaires

  • Docker Compose
  • Podman
  • Buildah
  • K3s
  • Helm

Conclusion

La technologie des conteneurs permet de moderniser les processus de développement et de déploiement avec efficacité, portabilité et sécurité. Grâce aux outils comme Docker et Kubernetes, elle constitue aujourd’hui un pilier essentiel du DevOps et du Cloud Computing.

3. Serverless Computing

Le serverless computing est un modèle d'exécution où les développeurs peuvent exécuter du code sans gérer l'infrastructure serveur sous-jacente. Le fournisseur cloud prend en charge la gestion, la scalabilité et la maintenance des serveurs.

Concepts clés

  • Fonctions en tant que service (FaaS) : exécution de fonctions indépendantes, déclenchées par des événements.
  • Backend en tant que service (BaaS) : services cloud prêts à l’emploi (base de données, authentification, stockage, etc.).
  • Événements déclencheurs : HTTP, messages, fichiers, timers...
  • Facturation à l’usage : paiement basé sur la durée d’exécution et les ressources consommées.

Avantages du Serverless

  • Pas de gestion de serveur
  • Scalabilité automatique
  • Coût réduit (pay-as-you-go)
  • Déploiement rapide
  • Focus sur le code métier

Inconvénients

  • Dépendance au fournisseur cloud
  • Limites d’exécution (timeout, ressources)
  • Difficulté à gérer les états persistants
  • Cold start : délai de démarrage lors de la première invocation
  • Moins de contrôle sur l’environnement

Principaux fournisseurs Serverless

  • AWS Lambda
  • Azure Functions
  • Google Cloud Functions
  • IBM Cloud Functions
  • Oracle Functions

Fonctionnement d’une fonction serverless

// Exemple simple d'une fonction AWS Lambda en Node.js
exports.handler = async (event) => {
  console.log("Event reçu :", event);
  return {
    statusCode: 200,
    body: JSON.stringify({ message: "Hello Serverless!" }),
  };
};

Modèles de déclencheurs

  • Requêtes HTTP via API Gateway
  • Événements de stockage (ex : upload sur S3)
  • Messages dans des files d’attente (ex : SNS, SQS)
  • Événements planifiés (cron jobs)
  • Modifications de base de données

Serverless vs Containers

Critère Serverless Conteneurs
Gestion serveur Aucune gestion, tout est abstrait Gestion partielle, notamment runtime
Scalabilité Automatique, instantanée Automatique mais parfois manuelle
Durée d’exécution Éphémère, limité (ex : 15 min max) Persistante
Coût Pay-as-you-go (exécution uniquement) Coût constant (instance/serveur)
Contrôle Restreint Complet

Cas d’usage courants

  • APIs REST / Backend léger
  • Traitement d’événements (ex : IoT, logs)
  • Automatisation et tâches planifiées
  • Microservices éphémères
  • Prototypage rapide

Bonnes pratiques

  • Éviter les fonctions trop longues
  • Minimiser les dépendances
  • Gérer la configuration via des variables d’environnement
  • Surveiller les logs et métriques
  • Tester localement avec des simulateurs
  • Prévoir la gestion des erreurs et des timeout

Outils et frameworks serverless

  • Serverless Framework
  • AWS SAM (Serverless Application Model)
  • Azure Durable Functions
  • Google Cloud Build & Deploy
  • OpenFaaS

Conclusion

Le serverless computing facilite le développement rapide d’applications évolutives sans se soucier de l’infrastructure. Bien qu’il présente des limites et une dépendance fournisseur, il reste un paradigme clé du cloud moderne, complémentaire aux conteneurs et aux architectures microservices.

4. Cloud Computing Threats

Le cloud computing offre de nombreux avantages, mais expose aussi à des risques spécifiques liés à la sécurité des données, des applications et des infrastructures. Il est important de comprendre ces menaces pour mieux s’en protéger.

Principales menaces dans le cloud

  • Fuite de données : exposition accidentelle ou malveillante des données sensibles.
  • Perte de données : suppression ou corruption des données sans possibilité de récupération.
  • Compte compromis : accès non autorisé via le piratage des identifiants ou des clés API.
  • Attaques par déni de service (DDoS) : surcharge des ressources pour rendre le service indisponible.
  • Malwares et ransomwares : logiciels malveillants visant à voler, corrompre ou bloquer l’accès aux données.
  • Failles dans les API cloud : vulnérabilités dans les interfaces de programmation utilisées pour gérer les services.
  • Violations de conformité : non-respect des réglementations liées à la protection des données.
  • Insider threats : menaces venant des employés ou partenaires malveillants ou négligents.
  • Configuration incorrecte : mauvaise gestion des accès et paramètres de sécurité exposant les ressources.

Vecteurs d’attaque courants

  • Phishing ciblant les identifiants cloud
  • Exploitation des vulnérabilités logicielles
  • Attaques sur les services d’authentification (ex : MFA contourné)
  • Utilisation d’images ou scripts malveillants dans les environnements cloud
  • Escalade de privilèges via des failles dans les configurations

Conséquences possibles

  • Perte financière et interruption d’activité
  • Atteinte à la réputation
  • Sanctions légales pour non-conformité
  • Vol de propriété intellectuelle
  • Accès non autorisé à des informations confidentielles

Mesures de prévention

  • Mettre en place une gestion rigoureuse des accès (IAM)
  • Utiliser l’authentification multi-facteurs (MFA)
  • Chiffrer les données au repos et en transit
  • Surveiller les logs et les activités suspectes
  • Appliquer des correctifs de sécurité régulièrement
  • Former les utilisateurs aux bonnes pratiques de sécurité
  • Utiliser des outils de détection et prévention des intrusions
  • Configurer correctement les services cloud et éviter les configurations par défaut

Normes et réglementations importantes

  • RGPD : protection des données personnelles en Europe
  • ISO 27001 : norme internationale sur la sécurité de l’information
  • PCI DSS : sécurisation des données de carte bancaire
  • HIPAA : protection des données de santé aux États-Unis

Bonnes pratiques

  • Auditer régulièrement la sécurité du cloud
  • Segmenter les réseaux et isoler les environnements
  • Mettre en place des politiques de sauvegarde et de restauration
  • Adopter le principe du moindre privilège
  • Utiliser des services cloud reconnus et bien sécurisés

Outils et services pour la sécurité cloud

  • Solutions CASB (Cloud Access Security Broker)
  • Firewall applicatif cloud (WAF)
  • Outils de gestion des identités (IAM)
  • Services de chiffrement et gestion des clés
  • Solutions SIEM pour la surveillance et l’analyse des logs

Conclusion

La sécurité dans le cloud nécessite une approche globale incluant des mesures techniques, organisationnelles et humaines. La connaissance des menaces et la mise en place de contrôles adaptés sont essentielles pour protéger les données et assurer la confiance dans les services cloud.

5. Cloud Hacking

Cloud Hacking désigne l’ensemble des techniques et attaques visant les infrastructures, services et applications hébergés dans le cloud. Avec l’adoption massive du cloud computing, la sécurité des environnements cloud est devenue cruciale.

Qu’est-ce que le Cloud ?

Le cloud computing permet d’accéder à des ressources informatiques (serveurs, stockage, bases de données, logiciels) via Internet, fournies par des fournisseurs comme AWS, Azure, Google Cloud.

Enjeux de sécurité dans le Cloud

  • Confidentialité des données
  • Intégrité des services
  • Disponibilité des ressources
  • Gestion des accès et authentification
  • Surveillance et détection d’intrusions

Principaux vecteurs d’attaque en Cloud Hacking

  • Exploitation de vulnérabilités dans les applications cloud ou l’infrastructure (ex : failles non corrigées).
  • Vol de clés d’API ou de secrets d’authentification exposés.
  • Attaques par force brute sur les comptes utilisateurs et administrateurs.
  • Attaques par injection (SQL, NoSQL, commandes).
  • Escalade de privilèges via des failles de configuration.
  • Attaques de type Man-in-the-Middle sur les communications non chiffrées.
  • Phishing et ingénierie sociale ciblant les employés ou administrateurs cloud.
  • Exploitation de mauvaises configurations (ex : buckets S3 publics, règles firewall laxistes).

Techniques courantes en Cloud Hacking

  • Reconnaissance : collecte d’informations sur les ressources cloud et configurations (ex : scan de ports, énumération de services).
  • Exploitation des failles : injection, cross-site scripting, vulnérabilités dans les API cloud.
  • Escalade de privilèges : exploitation des rôles mal configurés, failles IAM (Identity and Access Management).
  • Pivoting : déplacement latéral vers d’autres ressources dans le cloud.
  • Persistence : implantation de backdoors ou accès prolongé via clés compromises.

Principales vulnérabilités et erreurs fréquentes

  • Buckets de stockage publics (ex : Amazon S3)
  • Mauvaises configurations IAM
  • Utilisation de clés d’accès dans le code source public
  • Absence de chiffrement des données sensibles
  • Manque de surveillance des logs et alertes
  • Mauvaise gestion des versions des services et correctifs
  • Interfaces d’administration accessibles sans restrictions

Outils utilisés pour le Cloud Hacking

  • ScoutSuite : audit de configurations cloud
  • Prowler : vérification de la sécurité AWS
  • CloudSploit : scanner de vulnérabilités cloud
  • PacBot : plateforme d’analyse et de conformité cloud
  • AWS CLI : pour interagir et tester les configurations
  • Burp Suite : test des API et applications web cloud
  • Nmap : scan de réseaux cloud
  • Terraform Validator : analyse de la sécurité des infrastructures en tant que code

Principales plateformes cloud ciblées

  • Amazon Web Services (AWS)
  • Microsoft Azure
  • Google Cloud Platform (GCP)
  • IBM Cloud
  • Oracle Cloud

Bonnes pratiques pour sécuriser le cloud

  • Appliquer le principe du moindre privilège pour les comptes et rôles IAM
  • Activer l’authentification multi-facteur (MFA)
  • Surveiller et analyser régulièrement les logs d’accès
  • Mettre en place un chiffrement fort des données au repos et en transit
  • Effectuer des audits réguliers des configurations
  • Éviter de stocker des secrets dans le code source
  • Utiliser des outils de gestion de la conformité (CSPM)
  • Segmenter les réseaux et limiter les accès externes

Exemple d’attaque simple : Bucket S3 public

// Exemple de commande pour tester un bucket S3 public avec AWS CLI
aws s3 ls s3://nom-du-bucket --no-sign-request

// Si la liste des fichiers est affichée, le bucket est public et vulnérable

Exemple d’attaque IAM mal configuré

Un utilisateur ayant trop de privilèges (admin) peut exploiter des fonctions cloud pour prendre le contrôle total de l’infrastructure.

Impact du Cloud Hacking

  • Fuite massive de données confidentielles
  • Interruption de service ou déni de service
  • Atteinte à la réputation de l’entreprise
  • Perte financière (rançon, sanctions)
  • Compromission de la chaîne logistique logicielle

Ressources et formations recommandées

  • AWS Security Documentation
  • Azure Security Documentation
  • Google Cloud Security
  • Cloud Security Alliance (CSA)
  • Formations sécurité AWS (Pluralsight)
  • Cours Cloud Security sur Udemy

Conclusion

Le cloud hacking est un domaine en pleine expansion qui nécessite une vigilance constante, une bonne compréhension des architectures cloud et des pratiques de sécurité adaptées. La sécurisation du cloud repose sur la combinaison d’outils, de bonnes configurations, d’une surveillance active et d’une sensibilisation des équipes.

🔐5- Cloud Security Overview

Cloud security fait référence à l’ensemble des politiques, des technologies et des contrôles mis en œuvre pour protéger les environnements, les applications et les données dans le cloud.

🛡️ Sécurité assurée par les fournisseurs de Cloud

  • Contrôle d'accès strict et politiques d’autorisation
  • Journalisation des accès pour la traçabilité
  • Obligation de justification pour certains accès sensibles (non-répudiation)

🔧 Trusted Computing (TC)

La Trusted Computing repose sur des composants matériels renforçant la sécurité au niveau de l'ordinateur. Elle repose sur des Roots of Trust (RoT), un ensemble de fonctions au sein du Trusted Computing Module (TCM) toujours considérées comme fiables par le système d'exploitation.

⚠️ Principaux risques liés au Cloud

  • Vol de données : Interne ou externe, souvent silencieux (numéros de carte, numéros de sécurité sociale...)
  • Perte de données : Via malware ou suppression accidentelle. ➡️ Sauvegardes indispensables !
  • Utilisation abusive de la puissance de calcul : Pour casser des mots de passe ou lancer des attaques DDoS.
  • Shadow IT : Utilisation de solutions IT non validées officiellement (souvent non sécurisées)
  • Interfaces et API non sécurisées : Authentification faible ou mauvaise configuration
  • Technologie partagée mal isolée : Risques dans les environnements multi-tenant (ex. compromission de l’hyperviseur)
  • Faible visibilité sur le profil de risque : L’utilisateur ne sait pas exactement quelles protections sont mises en place.
  • Attaques via erreurs d’infrastructure ou conception inadéquate
  • Authentification faible : Risques d’escalade de privilèges ou de compromission croisée (ex: SAML forgé)
  • Risques de conformité : Ex. transfert de données transfrontaliers non autorisés
  • Cloud Cryptojacking : Usage non autorisé de ressources cloud pour miner de la crypto

🧨 Types d'attaques spécifiques au Cloud

  • Attaques d’ingénierie sociale : Phishing, devinette de mots de passe
  • XSS & SQLi : Injections dans les applications web déployées dans le cloud
  • DNS attacks : empoisonnement, détournement de domaines
  • Sniffing réseau & détournement de session
  • Cryptanalyse : Faiblesses dans le chiffrement
  • DoS/DDoS : via infrastructures cloud massives

🧬 Attaques avancées et ciblées

📦 Wrapping Attack (XML rewriting)

Interception d’un message SOAP et modification de sa structure sans invalider la signature XML. Cela peut contourner l’authentification dans les services web.

🎯 Session Riding (CSRF)

Le pirate utilise les cookies d’un utilisateur connecté pour envoyer des requêtes en son nom. Cela fonctionne également dans les environnements cloud.

🔍 Side Channel Attacks (Cross-VM Breach)

Exploitation des ressources partagées dans les hyperviseurs pour extraire des données sensibles d'une autre VM. Ex: cache CPU, mémoire RAM partagée.

🌩️ Cloud Hopper (APT ciblée)

Opération menée par des groupes comme APT10 pour infiltrer des fournisseurs de services cloud (MSPs), accéder aux réseaux clients et voler des données.

🧬 Cloudborne Attack

Exploitation du firmware BMC (Baseboard Management Controller) dans des serveurs bare-metal. Peut injecter des backdoors, provoquer des PDoS, etc.

🕵️‍♂️ Man-In-The-Cloud (MITC)

Usage malveillant des services de synchronisation comme Google Drive ou Dropbox pour exfiltrer des données sans traces sur l’appareil (utilisation de tokens détournés).

🛠️ Outils de sécurité Cloud recommandés

  • CloudInspect : Outil de pentesting pour Amazon EC2
  • CloudPassage Halo : Automatisation des contrôles de sécurité et conformité
  • privacy.sexy : Réduction de la collecte de données et durcissement OS/cloud

6. Containers

Les conteneurs permettent d'exécuter plusieurs applications de manière isolée sur un même système hôte, optimisant ainsi l'utilisation des ressources. Docker est devenu la norme pour leur exécution.

Containers vs Machines virtuelles

  • Les deux réduisent les coûts grâce au matériel partagé (un seul serveur physique peut héberger plusieurs VMs ou conteneurs).
  • Une machine virtuelle virtualise un système d'exploitation complet, tandis qu’un conteneur isole uniquement une application.

Menaces liées aux containers

Modèle de menace d’un container

  • Évasion du container : L'attaquant prend le contrôle d'une application, puis tente d'exploiter l'hôte via des failles système ou noyau.
  • Attaque d'autres containers via le réseau : L'attaquant tente de compromettre d'autres conteneurs.
  • Attaque de l'orchestrateur : Failles dans les interfaces de gestion comme Kubernetes (souvent mal sécurisées par défaut).
  • Compromission de l'hôte via le réseau : Ouvrir des ports ou exploiter des services non sécurisés.
  • Attaques sur des ressources partagées : LDAP, Jenkins, systèmes de fichiers réseaux ou sniffing réseau.
  • Épuisement des ressources : CPU, mémoire, I/O, etc. provoquant un déni de service.
  • Compromission de l'hôte : via le réseau ou un autre container.
  • Intégrité des images : L’attaquant injecte des charges malveillantes pendant le cycle CI/CD.

Top 10 OWASP Docker

  1. Mapping utilisateur sécurisé : Docker s'exécute avec les privilèges root.
  2. Gestion des correctifs : Patcher les outils d’orchestration, OS et images.
  3. Ségrégation réseau : Sécuriser interfaces de gestion et services.
  4. Configuration sécurisée : Ne pas inclure de composants inutiles.
  5. Contexte de sécurité : Ne pas mélanger des containers sensibles avec d'autres moins sécurisés.
  6. Protection des secrets : Gérer mots de passe, tokens, clés privées.
  7. Protection des ressources : Empêcher qu’un conteneur monopolise les ressources.
  8. Intégrité des images : Vérification à chaque étape du déploiement.
  9. Principe d’immutabilité : Conteneurs en lecture seule si possible.
  10. Journalisation : Centraliser les logs avec horodatage et protection contre les altérations.

Exemples d'attaques sur les containers

  • Images vulnérables : Logiciels obsolètes dans l'image (ex : Apache OpenWhisk).
  • Docker non patché : Failles connues telles que :
    • CVE-2019-5736 : accès au système hôte.
    • CVE-2018-8115 : exécution distante de code.
    • CVE-2018-9862 : élévation de privilèges.
  • Conteneurs malveillants : (ex : crypto-mining près d'une zone nucléaire).
  • Outils d’orchestration compromis : Kubernetes, OpenShift, etc. (liste CVEs Kubernetes).

Avantages des containers face aux VMs

  • Pas d’accès SSH par défaut → Moins de vecteurs d'attaque.
  • Pas d’accès utilisateur prévu → Moins de risques de fuites d’identifiants.
  • Ports restreints par défaut.
  • Durée de vie courte → Attaques plus difficiles à réussir.
  • Modèle immutable → Plus difficile d’injecter des malwares.
  • Pipelines CI/CD automatisés → Mise à jour rapide.
  • APIs bien définies → Détection facile d’anomalies comportementales.

Contre-mesures de sécurité

  • OWASP Docker Security Cheat Sheet :
    • Mettre à jour l'hôte et Docker régulièrement
    • Ne pas exposer le socket Docker
    • Utiliser un utilisateur dédié
    • Limiter les privilèges avec --no-new-privileges
    • Désactiver la communication entre conteneurs : --icc=false
    • Activer les modules de sécurité Linux (seccomp, AppArmor, SELinux)
    • Limiter l'utilisation CPU, mémoire, processus, redémarrages, etc.
    • Monter les volumes en lecture seule
    • Analyser les images statiquement et dynamiquement
  • Validation du code source et des dépendances avant déploiement
  • Validation d'authenticité des images
  • Scan actif des conteneurs déployés
  • Inspection du trafic réseau (service firewall applicatif)
  • Centralisation des logs
  • Injection dynamique de secrets (just-in-time)
  • Vérification régulière des conteneurs en cours d’exécution
  • Utilisation d’images à jour
  • Ne jamais exécuter en root : rootless mode
